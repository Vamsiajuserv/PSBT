# EO Command Center — Behavioral Snapshots

Each scenario pairs the **backend JSON** (`snapshots/<scenario>.json`, generated by
`test_command_dashboard.py`) with the **expected frontend behavior**. Together they are the
regression contract for both layers. Regenerate JSON with `python tests/test_command_dashboard.py`.

Legend: ✅ provider healthy · ⚠️ degraded · ❌ unavailable

---

## Scenario: `normal`
| Provider | Status |
|---|---|
| Operations | ✅ |
| Attention | ✅ (3 items) |
| Financial | ✅ |
| Activity | ✅ |

**Expected UI**
- All 4 widgets render `fresh`; no stale badges.
- Attention Queue shows items ordered by severity (HIGH first), each with a deep-link action.
- KPIStrip shows today's booking counts + revenue.
- No partial-data banner. `health.completenessPct = 100`.
- No React errors.

## Scenario: `partial_failure`  (Financial source down)
| Provider | Status |
|---|---|
| Operations | ✅ |
| Attention | ✅ |
| Financial | ❌ |
| Activity | ✅ |

**Expected UI**
- **Financial widget shows "Data unavailable"** with a Retry button (WidgetFrame `unavailable`).
- The **other 3 widgets still render** normally — one dead provider must not blank the dashboard.
- **Partial-data banner visible** at the top (`payload.partial = true`).
- `health.completenessPct = 75`, `degraded = 1`.
- No React errors; no full-page failure.

## Scenario: `high_alert`  (festival day / incident)
| Provider | Status |
|---|---|
| Operations | ✅ |
| Attention | ✅ (7 items incl. 2 CRITICAL) |
| Financial | ✅ (settlement PENDING_APPROVAL) |
| Activity | ✅ |

**Expected UI**
- Attention Queue is the visual focus: CRITICAL items first with the strongest severity accent.
- Attention count is high but the list scrolls within its widget (does not push the layout).
- Financial widget surfaces the PENDING settlement.
- Dashboard still composes at `completenessPct = 100` (all providers healthy, just busy).
- **Success test:** can the EO spot the two CRITICAL items and act within ~5 seconds?

## Invariant contracts (enforced by `test_command_dashboard.py`)
These encode the ADRs as executable architecture tests:
1. A provider failure **never** fails the dashboard (partial → 200 with degraded widget).
2. Every widget carries a **confidence envelope** (`meta` with all keys).
3. Every widget carries a **version**.
4. Providers return **domain data only** (no `icon`/`color`/`className`/`style`).
5. Composition is **JSON-serializable** — no router/ORM DTO leaks through.
6. `health.completenessPct` reflects degraded providers (100 normal, <100 partial).
